- `/pr-review create` — Create a new pull request

## Instructions

### `/pr-review create`

#### Step 1: Analyze Changes
```bash
# Get current branch
BRANCH=$(git branch --show-current)

# See all changes
git diff main...HEAD --stat
git diff main...HEAD --name-only

# Get commit history
git log main..HEAD --oneline
```

#### Step 2: Identify Related User Story
Look for user story references in:
- Branch name (e.g., `feature/US-001-user-registration`)
- Commit messages
- Sprint plan documents

#### Step 3: Create PR

```bash
gh pr create \
  --title "[US-XXX] Short description of changes" \
  --body "$(cat <<'EOF'
## Summary
- [Bullet point 1: What was implemented]
- [Bullet point 2: Key technical decisions]
- [Bullet point 3: Notable changes]

## Related Issues
- Closes #[issue-number] ([US-XXX] User Story Title)
- Related to #[epic-number] ([EPIC-XXX] Epic Title)

## Changes Made

### Backend
- [List of backend changes with file paths]

### Frontend
- [List of frontend changes with file paths]

### Database
- [Migration files added]

## Testing
- [ ] Unit tests added/updated (coverage: X%)
- [ ] Integration tests pass
- [ ] E2E tests pass (if applicable)
- [ ] Manual testing completed

## Security Checklist
- [ ] No hardcoded secrets
- [ ] Input validation on all endpoints
- [ ] Authentication/authorization verified
- [ ] No SQL injection vulnerabilities
- [ ] XSS prevention in place

## Screenshots
[If UI changes, add before/after screenshots]

## Deployment Notes
[Any special deployment considerations]

---
Generated by AI-Native SDLC Factory
EOF
)" \
  --base main
```

#### Step 4: Link to Project Board
```bash
# Get PR number from creation output
PR_NUMBER=$(gh pr view --json number -q .number)

# Add to project board if configured
PROJECT_NUM=$(cat .sdlc/state.json | python3 -c "import sys,json; print(json.load(sys.stdin).get('github',{}).get('projectNumber',''))" 2>/dev/null)
if [ -n "$PROJECT_NUM" ]; then
  gh project item-add "$PROJECT_NUM" --url "$(gh pr view --json url -q .url)"
fi
```

- Set `phases.code_review.status` to `"in_progress"`

### `/pr-review review [pr-number]`

#### Step 1: Fetch PR Details
```bash
gh pr view $2 --json title,body,files,additions,deletions,changedFiles
gh pr diff $2
```

#### Step 2: Spawn Explore Agent for Code Analysis
Use the Task tool to spawn an **Explore** subagent:

```
Analyze this PR diff for the following issues:

1. **Architecture Compliance**: Does the code follow DDD layered architecture?
   - Domain entities have business logic (not anemic)?
   - Services are in correct layer?
   - No domain logic in controllers?

2. **Design Patterns**: Are patterns used correctly?
   - Builder for complex objects?
   - Repository pattern for data access?
   - DTOs separate from domain entities?

3. **Security Issues** (OWASP Top 10):
   - SQL injection (string concatenation in queries)?
   - XSS (unescaped output)?
   - Hardcoded credentials/tokens?
   - Missing input validation?
   - Missing authorization checks?

4. **Code Quality**:
   - Methods too long (>30 lines)?
   - Missing error handling?
   - Unused imports/variables?
   - Missing null checks?
   - Proper logging?

5. **Testing**:
   - Are there tests for new code?
   - Do tests cover edge cases?
   - Are mocks used correctly?

6. **Best Practices**:
   - Proper HTTP status codes?
   - Consistent naming?
   - JavaDoc/JSDoc on public APIs?
```

### `/pr-review merge [pr-number]`

Merge an approved PR and sync local branches. This is the critical post-review step that closes the loop.

#### Step 1: Verify Merge Readiness
```bash
# Check PR status — must be approved with passing checks
gh pr view $2 --json state,reviewDecision,statusCheckRollup \
  --jq '{state: .state, decision: .reviewDecision, checks: [.statusCheckRollup[]?.conclusion]}'
```

If the PR is not approved or CI checks are failing, **STOP** and report the issue. Do not merge.

#### Step 2: Squash Merge to main
```bash
# Squash merge (matches GitHub Flow strategy from 01-general.md)
# --delete-branch removes the remote feature branch automatically
gh pr merge $2 --squash --delete-branch
```

#### Step 3: Sync Local main
```bash
# Switch to main and pull the merged changes
git checkout main
git fetch origin
git pull origin main
```

#### Step 4: Clean Up Local Feature Branch
```bash
# Delete the local feature branch (the one that was merged)
FEATURE_BRANCH=$(gh pr view $2 --json headRefName -q .headRefName)
git branch -d "$FEATURE_BRANCH" 2>/dev/null || echo "Local branch already deleted or not found"
```

#### Step 5: Verify Sync
```bash
# Confirm local main matches remote
LOCAL_SHA=$(git rev-parse main)
REMOTE_SHA=$(git rev-parse origin/main)

if [ "$LOCAL_SHA" = "$REMOTE_SHA" ]; then
  echo "Local main is in sync with remote."
else
  echo "WARNING: Local main differs from remote. Run: git pull origin main"
fi
```

#### Step 6: Update SDLC State
- Add merge event to `phases.code_review.artifacts` in `.sdlc/state.json`
- If all sprint PRs are merged, set `phases.code_review.status` to `"completed"`
- Update `updatedAt`

```

PR: #[number] — [title]
Merged to: main (squash merge)
Remote branch: deleted
Local branch: deleted
Local main: synced

Sync Status:
  Local main SHA:  [sha]
  Remote main SHA: [sha]
  Match: YES

Next Steps:
1. Run /sdlc next to advance to Release phase
2. Or /pr-review list to check remaining PRs
3. Or /develop [layer] [next-story] to continue development
```

## Multi-Stack PR Review

### All-Workspace PR Template
Read `.claude/rules/06-tech-stack-context.md` and `.sdlc/state.json → techStack`.

When creating PRs for multi-stack projects, the PR body should include sections for ALL affected workspaces:

```markdown
### Backend (Spring Boot)
- [Changes in backend/]

### Frontend (Angular)
- [Changes in frontend/]

### [Workspace Name] ([Technology])
- [Changes in workspaces/<dir>/]
```

### Cross-Workspace Review
When reviewing PRs that touch multiple workspaces:
- Verify shared contract compliance (check `docs/tech-specs/shared-schemas/`)
- Ensure cross-workspace integration points are not broken
- Check that workspace-specific conventions are followed (consult `docs/tech-refs/<workspace>/`)
- Use workspace-specific build/test commands from state.json to verify

### `/pr-review merge [N]` → DevOps Agent
- **Agent**: `.claude/agents/devops-agent.md` (Haiku model)

- Squash-merges PR to main, syncs local main, deletes feature branch
- Fast execution (Haiku model)

### `/pr-review review [N]` → Review Agent + Validator Agent (parallel)
- **Review Agent**: `.claude/agents/review-agent.md` (Opus model, read-only)
  - PR diff analysis, architecture compliance, code quality, security patterns
  - Verdict: APPROVE / REQUEST CHANGES / COMMENT
- **Validator Agent**: `.claude/agents/validator-agent.md` (Opus model, read-only)
  - DDD governance, boundary violations, coupling score, DDD Health Score
  - Verdict: PASS / BLOCK

Both agents are **read-only** and can run in **parallel** (Agent Teams recommended).

### When using Agent Teams (recommended for review):

```
│ → APPROVE/REJECT │ → PASS/BLOCK     │
└──────────────────┴──────────────────┘
```
