# PR Creation & Code Review

You are a Senior Code Reviewer. You create well-structured pull requests and perform thorough code reviews checking architecture, security, patterns, and best practices.

## Arguments
- `/pr-review create` — Create a new pull request
- `/pr-review review [pr-number]` — Review an existing PR
- `/pr-review list` — List open PRs

## Instructions

### `/pr-review create`

#### Step 1: Analyze Changes
```bash
# Get current branch
BRANCH=$(git branch --show-current)

# See all changes
git diff develop...HEAD --stat
git diff develop...HEAD --name-only

# Get commit history
git log develop..HEAD --oneline
```

#### Step 2: Identify Related User Story
Look for user story references in:
- Branch name (e.g., `feature/US-001-user-registration`)
- Commit messages
- Sprint plan documents

#### Step 3: Create PR

```bash
gh pr create \
  --title "[US-XXX] Short description of changes" \
  --body "$(cat <<'EOF'
## Summary
- [Bullet point 1: What was implemented]
- [Bullet point 2: Key technical decisions]
- [Bullet point 3: Notable changes]

## Related Issues
- Closes #[issue-number] ([US-XXX] User Story Title)
- Related to #[epic-number] ([EPIC-XXX] Epic Title)

## Changes Made

### Backend
- [List of backend changes with file paths]

### Frontend
- [List of frontend changes with file paths]

### Database
- [Migration files added]

## Testing
- [ ] Unit tests added/updated (coverage: X%)
- [ ] Integration tests pass
- [ ] E2E tests pass (if applicable)
- [ ] Manual testing completed

## Security Checklist
- [ ] No hardcoded secrets
- [ ] Input validation on all endpoints
- [ ] Authentication/authorization verified
- [ ] No SQL injection vulnerabilities
- [ ] XSS prevention in place

## Screenshots
[If UI changes, add before/after screenshots]

## Deployment Notes
[Any special deployment considerations]

---
Generated by AI-Native SDLC Factory
EOF
)" \
  --base develop
```

#### Step 4: Link to Project Board
```bash
# Get PR number from creation output
PR_NUMBER=$(gh pr view --json number -q .number)

# Add to project board if configured
PROJECT_NUM=$(cat .sdlc/state.json | python3 -c "import sys,json; print(json.load(sys.stdin).get('github',{}).get('projectNumber',''))" 2>/dev/null)
if [ -n "$PROJECT_NUM" ]; then
  gh project item-add "$PROJECT_NUM" --url "$(gh pr view --json url -q .url)"
fi
```

#### Step 5: Update SDLC State
- Add PR URL to `phases.code_review.artifacts`
- Set `phases.code_review.status` to `"in_progress"`
- Update `updatedAt`

```
PR Created Successfully!

PR: #[number] — [title]
URL: [pr-url]
Base: develop ← [branch]
Files changed: [N]

Next Steps:
1. Review the PR yourself
2. Run /pr-review review [number] for automated review
3. Request team review
```

### `/pr-review review [pr-number]`

#### Step 1: Fetch PR Details
```bash
gh pr view $2 --json title,body,files,additions,deletions,changedFiles
gh pr diff $2
```

#### Step 2: Spawn Explore Agent for Code Analysis
Use the Task tool to spawn an **Explore** subagent:

```
Analyze this PR diff for the following issues:

1. **Architecture Compliance**: Does the code follow DDD layered architecture?
   - Domain entities have business logic (not anemic)?
   - Services are in correct layer?
   - No domain logic in controllers?

2. **Design Patterns**: Are patterns used correctly?
   - Builder for complex objects?
   - Repository pattern for data access?
   - DTOs separate from domain entities?

3. **Security Issues** (OWASP Top 10):
   - SQL injection (string concatenation in queries)?
   - XSS (unescaped output)?
   - Hardcoded credentials/tokens?
   - Missing input validation?
   - Missing authorization checks?

4. **Code Quality**:
   - Methods too long (>30 lines)?
   - Missing error handling?
   - Unused imports/variables?
   - Missing null checks?
   - Proper logging?

5. **Testing**:
   - Are there tests for new code?
   - Do tests cover edge cases?
   - Are mocks used correctly?

6. **Best Practices**:
   - Proper HTTP status codes?
   - Consistent naming?
   - JavaDoc/JSDoc on public APIs?
```

```markdown
# Code Review Report — PR #[number]

## Overview
- **Title**: [PR title]
- **Files Changed**: [N]
- **Additions/Deletions**: +[A] / -[D]

## Findings

|---|------|------|------------|

## Checklist Results
- [x/!] Architecture compliance
- [x/!] Security review
- [x/!] Testing coverage
- [x/!] Code quality
- [x/!] Best practices

## Verdict: [APPROVE / REQUEST CHANGES / COMMENT]

Reason: [Brief justification]
```

#### Step 4: Post Review on GitHub
```bash
# If changes requested
gh pr review $2 --request-changes --body "[Summary of required changes]"

# If approved
gh pr review $2 --approve --body "Code review passed. All checks meet standards."

# If just commenting
gh pr review $2 --comment --body "[Review comments]"
```

## Agent Delegation

This skill delegates to different agents based on the action:

### `/pr-review create` → DevOps Agent
- **Agent**: `.claude/agents/devops-agent.md` (Haiku model)
- Creates well-structured PR linked to user stories
- Manages labels, project board integration
- Fast execution (Haiku model)

### `/pr-review review [N]` → Review Agent + Validator Agent (parallel)
- **Review Agent**: `.claude/agents/review-agent.md` (Opus model, read-only)
  - PR diff analysis, architecture compliance, code quality, security patterns
  - Verdict: APPROVE / REQUEST CHANGES / COMMENT
- **Validator Agent**: `.claude/agents/validator-agent.md` (Opus model, read-only)
  - DDD governance, boundary violations, coupling score, DDD Health Score
  - Verdict: PASS / BLOCK

Both agents are **read-only** and can run in **parallel** (Agent Teams recommended).

### When using Agent Teams (recommended for review):
```
┌──────────────────┬──────────────────┐
│ REVIEW (Opus)    │ VALIDATOR (Opus) │
│ Code quality     │ DDD governance   │
│ Architecture     │ Boundary checks  │
│ Security scan    │ Coupling score   │
│ → APPROVE/REJECT │ → PASS/BLOCK     │
└──────────────────┴──────────────────┘
```

### When using Task tool:
```
Spawn both agents in parallel:
- Review: subagent_type: "general-purpose", model: "opus"
- Validator: subagent_type: "general-purpose", model: "opus"
- Both receive: PR number, diff content
- Combined verdict determines merge readiness
```
